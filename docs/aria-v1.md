# ARIA V1 вЂ” Proactive Intelligence Layer

ARIA (Adaptive Residential Intelligence Assistant) is the AI-powered intelligence layer built into Cornerstone. V1 delivers proactive suggestion generation, human approval workflow, management oversight drafting, linked record intelligence, and a full audit trail.

ARIA strengthens professional judgement вЂ” it does not replace it. Every suggestion requires human review before any action is taken.

## Database Schema

### aria_suggestions

Core table for proactive suggestions. Each row represents one suggestion generated by ARIA.

| Column | Type | Description |
|--------|------|-------------|
| id | text PK | Prefixed `as_` |
| organisation_id | text | Scoping |
| home_id | text | Scoping |
| child_id | text | Which child this relates to |
| staff_id | text | Which staff member (if applicable) |
| related_record_type | text NOT NULL | Source record type (e.g. `incident`) |
| related_record_id | text | Source record ID |
| suggestion_type | text NOT NULL | One of 12 types (see below) |
| title | text NOT NULL | Human-readable title |
| summary | text | What the suggestion is about |
| reason | text | Why ARIA is suggesting this |
| suggested_action | text | What the manager should do |
| draft_text | text | AI-generated draft content |
| final_text | text | Manager-approved text |
| risk_level | text NOT NULL | `urgent` / `high` / `medium` / `low` |
| confidence_level | text NOT NULL | `low` / `medium` / `high` |
| status | text NOT NULL | Workflow status (see lifecycle below) |
| requires_approval | boolean NOT NULL | Always `true` in V1 |
| reviewer_role | text | Expected reviewer role |
| created_by | text | Actor who triggered generation |
| reviewed_by / approved_by / rejected_by / committed_by | text | Workflow actors |
| rejection_reason | text | Required when status = `rejected` |
| ai_provider | text | Which AI provider was used |
| mock_mode | boolean NOT NULL | `true` if no AI API key configured |
| created_at / reviewed_at / approved_at / rejected_at / committed_at / updated_at | timestamptz | Timestamps |

Indexes: status, related_record_type, related_record_id, child_id, home_id, risk_level, suggestion_type, created_at.

### aria_suggestion_links

Linked record suggestions per suggestion. When ARIA identifies that an incident may affect other records (risk assessments, behaviour support plans, etc.), those are stored here.

| Column | Type | Description |
|--------|------|-------------|
| id | text PK | Prefixed `asl_` |
| suggestion_id | text FK | References aria_suggestions |
| linked_record_type | text NOT NULL | Target record type |
| linked_record_id | text | Target record ID (if known) |
| relationship_type | text | How records are related |
| reason | text | Why this record is linked |
| suggested_action | text | What to do with this record |
| risk_level | text | Risk level for this link |
| requires_approval | boolean | Default `true` |

### aria_suggestion_audit

Append-only audit trail. Every action on a suggestion is logged.

| Column | Type | Description |
|--------|------|-------------|
| id | text PK | Prefixed `asa_` |
| suggestion_id | text FK | References aria_suggestions |
| actor_user_id | text | Who performed the action |
| actor_role | text | Their role at time of action |
| action | text NOT NULL | One of 13 audit actions |
| before_state / after_state | jsonb | State snapshots |
| metadata | jsonb | Additional context |

### Row Level Security

- `service_role`: full access (read + write)
- `authenticated`: read-only

## Approval Lifecycle

```
draft в”Ђв”Ђв–¶ awaiting_review в”Ђв”Ђв”¬в”Ђв”Ђв–¶ approved в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–¶ committed
                            в”њв”Ђв”Ђв–¶ amended_and_approved в”Ђв”Ђв–¶ committed
                            в”њв”Ђв”Ђв–¶ rejected
                            в”њв”Ђв”Ђв–¶ no_action_required
                            в””в”Ђв”Ђв–¶ archived
```

- **approved**: Manager accepts the draft text as-is
- **amended_and_approved**: Manager edits the draft before approving (both versions stored)
- **rejected**: Manager rejects вЂ” rejection reason is mandatory
- **no_action_required**: Manager acknowledges but no action needed
- **committed**: Approved text has been written to the target record

## Service Layer

File: `src/lib/aria/aria-suggestions.ts`

### Core Functions

| Function | Purpose |
|----------|---------|
| `generateIncidentSuggestions(input)` | Generate suggestions from an incident. Returns typed suggestions based on severity |
| `generateManagementOversightDraft(input)` | Generate an oversight draft using LLM (or deterministic fallback) |
| `persistSuggestions(suggestions, actor)` | Save suggestions to database with audit entries |
| `getSuggestions(filters)` | Query suggestions with status/risk/type/child/home filters |
| `getSuggestionById(id, actor)` | Fetch single suggestion with linked records (logs view event) |
| `approveSuggestion(id, actor, finalText?)` | Approve вЂ” detects amended text automatically |
| `rejectSuggestion(id, actor, reason)` | Reject вЂ” reason is required (empty string returns null) |
| `markNoAction(id, actor)` | Mark as no action required |
| `commitSuggestion(id, actor, recordType?, recordId?)` | Commit approved text to a target record |
| `getAuditTimeline(suggestionId)` | Get chronological audit entries for a suggestion |
| `writeAuditEntry(args)` | Write a single audit entry |

### Mock Mode

When no AI API key is configured (`OPENAI_API_KEY` / `ANTHROPIC_API_KEY`), ARIA runs in mock mode:

- `generateIncidentSuggestions` uses deterministic templates вЂ” realistic professional content based on suggestion type, severity, and incident type
- `generateManagementOversightDraft` uses a structured deterministic template covering all oversight areas with manager-fillable sections
- All suggestions, audit logs, and queue items are still created вЂ” the workflow is identical
- `mock_mode: true` is set on each suggestion and visible in the UI

## API Routes

### GET /api/aria/suggestions

List suggestions with optional filters.

Query params: `actorUserId` (required), `actorRole`, `status`, `related_record_type`, `related_record_id`, `child_id`, `home_id`, `risk_level`

Permission: `aria.use`

### POST /api/aria/suggestions

Generate suggestions from an incident.

Body: `actorUserId` (required), `actorRole`, `incidentId`, `incidentType`, `severity`, `description`, `immediateAction`, `childId`, `staffId`, `homeId`, `organisationId`

Permission: `aria.generate_drafts`

### GET /api/aria/suggestions/[id]

Fetch a single suggestion with linked records. Logs a `suggestion_viewed` audit event.

Query params: `actorUserId` (required), `actorRole`

Permission: `aria.use`

### PATCH /api/aria/suggestions/[id]

Update suggestion status.

Body: `actorUserId` (required), `actorRole`, `action` (one of: `approve`, `reject`, `no_action`, `commit`), `finalText` (for approve), `rejectionReason` (required for reject), `committedRecordType`, `committedRecordId` (for commit)

Permissions: `aria.approve_outputs` (approve/no_action), `aria.reject_outputs` (reject), `aria.commit_to_records` (commit)

### GET /api/aria/suggestions/[id]/audit

Fetch audit timeline for a suggestion.

Query params: `actorUserId` (required), `actorRole`

Permission: `aria.view_audit_logs`

### POST /api/aria/suggestions/oversight-draft

Generate a management oversight draft for an incident.

Body: `actorUserId` (required), `actorRole`, `description` (required), `incidentId`, `incidentType`, `severity`, `immediateAction`, `childId`, `staffId`, `homeId`, `organisationId`

Permission: `aria.generate_drafts`

## Writing Standards

Every ARIA-generated draft includes:
- `ARIA_PROFESSIONAL_IDENTITY_PROMPT` вЂ” 40-year sector expert identity
- `ARIA_WRITING_STYLE_PROMPT` вЂ” professional tone, regulatory language, trauma-informed
- `applyAriaPostprocessor()` вЂ” post-generation quality pass

All drafts are labelled: "Aria suggested draft вЂ” requires manager review before saving."
