# Cara Intelligence Architecture

## How Cara delivers AI-grade intelligence without regular API calls

---

## The Core Idea

Cara (Adaptive Regulatory Intelligence Assistant) provides real-time intelligence across 300+ care home domains — safeguarding alerts, compliance metrics, risk analysis, and reflective insights — **without making a single LLM API call during normal operation**.

Every metric, every alert, and every intelligence insight displayed on the Manager Control Centre dashboard is generated by **deterministic pure functions** running in TypeScript. No tokens. No latency. No API costs. No rate limits.

**Cara suggests. Humans decide. Cara evidences.**

## The Three Pure Functions

Every domain service (318 services across child, staff, and home domains) exports three pure functions. These take an array of database rows as input and return structured intelligence as output. No side effects. No network calls.

### 1. `computeMetrics(rows)`

Takes all records for a domain and computes quantitative intelligence:

- **Counts**: total records, high-risk counts, non-compliant counts
- **Rates**: percentage calculations using `Math.round(value * 1000) / 10` for single-decimal precision
- **Averages**: numeric averages (temperatures, hours, satisfaction scores)
- **Unique counts**: distinct children, staff, locations, assessors

**Example** — Child Trafficking Risk:
```typescript
computeMetrics(rows) => {
  total_assessments: 6,
  high_risk_count: 1,        // High + Immediate combined
  immediate_count: 0,
  nrm_referral_rate: 33.3,   // % with NRM referral made
  first_responder_rate: 50.0,
  safety_plan_rate: 50.0,
  unique_children: 4,
  unique_assessors: 2
}
```

No AI involved. Pure arithmetic on structured data.

### 2. `computeAlerts(rows)`

Scans every record against domain-specific regulatory rules and returns prioritised alerts:

- **Critical**: Immediate safety risks (e.g., prohibited equipment still in use, child at immediate risk without safety plan)
- **High**: Regulatory breaches requiring urgent action (e.g., no police notification for high-risk trafficking case)
- **Medium**: Compliance gaps needing attention (e.g., training incomplete, review overdue)

Each alert includes a `record_id` linking back to the specific record that triggered it.

**Example** — Home Gas Safety:
```typescript
computeAlerts(rows) => [
  {
    type: "immediately_dangerous",
    severity: "critical",
    message: "Immediately Dangerous result on boiler at Kitchen — do not use",
    record_id: "uuid-123"
  },
  {
    type: "no_co_alarm",
    severity: "medium",
    message: "Carbon monoxide alarm not tested at Bedroom 2",
    record_id: "uuid-456"
  }
]
```

The rules are encoded as conditional logic, not LLM prompts. Every alert is traceable to a specific `if` statement in the source code.

### 3. `generateAriaInsights(rows)`

Produces three natural-language intelligence strings by composing the outputs of `computeMetrics` and `computeAlerts`:

1. **Summary** (colour-tagged `[red]`, `[sky]`, `[fuchsia]`, etc.) — headline statistics
2. **Priority** (tagged `[amber]`) — what needs attention now, drawn from alert counts and rates
3. **Reflective** (tagged `[reflect]`) — a question that prompts the manager to think critically

**Example** — Staff Whistleblowing:
```
[violet] 5 disclosures across 4 staff. Open: 2. Under investigation: 1.
[amber] 1 detriment reported. Protection rate 80.0%. Investigation rate 60.0%.
[reflect] Are staff confident to raise concerns? Is the policy genuinely accessible?
```

These strings are **template-composed from computed data**, not generated by an LLM. The branching logic (singular/plural, positive/negative paths, threshold-based messaging) is all in TypeScript.

## Why This Works

### 1. Deterministic and auditable

Every output can be traced to a specific code path. In a regulated care environment (Ofsted, CQC), this matters. An inspector can ask "why did Cara flag this?" and the answer is a line of code, not "the model decided."

### 2. Zero marginal cost

Once deployed, there are no per-query costs. A home with 1,000 records computes the same three functions as one with 10 records. No token counting, no billing surprises.

### 3. Zero latency

Pure functions execute in microseconds. The dashboard loads instantly because metrics, alerts, and insights are computed in-browser or on the server without waiting for an external API.

### 4. Works offline

If the Supabase connection drops, the last-fetched rows still produce full intelligence. No dependency on external AI services for core functionality.

### 5. Comprehensive test coverage

Every pure function is tested with 150-300+ unit tests per service (~34,000+ tests total). Metrics, alerts, and insights are verified for:
- Empty datasets
- Single records
- Multiple records with mixed values
- Boundary conditions (rates at 0%, 50%, 100%)
- All enum values
- Rounding precision
- Singular/plural grammar in insights

## The Data Pipeline

```
1. Staff member records an event (e.g., lift inspection)
         │
         ▼
2. Record saved to Supabase (cs_home_lift_equipment_safety)
         │
         ▼
3. RLS ensures tenant isolation (get_my_home_id())
         │
         ▼
4. Dashboard fetches rows via API route (/api/operations/home-lift-equipment-safety)
         │
         ▼
5. Service returns { ok: true, data: rows }
         │
         ▼
6. Dashboard card calls:
   - computeMetrics(rows)   → numbers for the 4 metric boxes
   - computeAlerts(rows)    → red/amber alert banners
   - generateAriaInsights() → 3 intelligence strings
         │
         ▼
7. Manager sees real-time intelligence. No API call made.
```

## Domain Coverage

The system covers 318 domain services across three pillars:

### Child (Safeguarding & Welfare)
Trafficking risk, modern slavery, radicalisation, FGM, forced marriage, honour-based abuse, online safety, substance misuse, gangs affiliation, bereavement support, criminal exploitation, sexual exploitation, nutrition monitoring, and many more.

### Staff (Compliance & HR)
DBS renewals, supervision, training, payroll compliance, overtime management, whistleblowing, lone working risk, professional registration, exit interviews, agency worker compliance, secondment management, sickness management, and many more.

### Home (Safety & Environment)
Fire risk, gas safety, electrical safety, water hygiene, asbestos, legionella, CCTV compliance, pest control, emergency lighting, lift equipment, radon testing, energy efficiency, accessibility, insurance compliance, and many more.

Each domain has:
- A Postgres table with RLS (323 migrations)
- A TypeScript service with pure intelligence functions
- A REST API route
- A themed dashboard card on the Manager Control Centre
- 150-300+ unit tests

## When LLM API Calls ARE Used (Optional Tier 2)

Cara does have an optional LLM layer for advanced features that go beyond deterministic analysis:

| Feature | What it does | Provider |
|---------|-------------|----------|
| Cara Voice of Child | Analyses free-text recordings for sentiment and themes | Anthropic Claude |
| Management Oversight | Generates narrative oversight summaries from structured data | Anthropic Claude |
| HR Letters | Drafts employment letters from templates + context | OpenAI / Anthropic |
| Document Intelligence | Extracts structured data from uploaded documents | Anthropic Claude |
| Cara Compose | Free-form AI assistant for care staff | Configurable |

These features are:
- **Gated** behind `AI_PROVIDER` environment variable (defaults to `stub` mode with demo responses)
- **Multi-provider** — supports OpenAI, Anthropic, and Gemini via an abstraction layer
- **Not required** for core dashboard intelligence
- **Gracefully degraded** — if no API key is configured, the system falls back to safe stub responses

## Configuration

### Core intelligence (always active, no config needed)
```env
# Just needs Supabase for data storage
NEXT_PUBLIC_SUPABASE_URL=your-url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-key
```

### Optional LLM features
```env
# Enable LLM-powered features (optional)
AI_PROVIDER=anthropic          # or "openai" or "gemini" or "stub"
ANTHROPIC_API_KEY=sk-ant-...   # only if AI_PROVIDER=anthropic
OPENAI_API_KEY=sk-...          # only if AI_PROVIDER=openai
GEMINI_API_KEY=...             # only if AI_PROVIDER=gemini
```

If `AI_PROVIDER` is not set or set to `stub`, all LLM features return safe demo responses. The core dashboard intelligence works regardless.

## Summary

| Aspect | Core Intelligence (Tier 1) | LLM Features (Tier 2) |
|--------|---------------------------|----------------------|
| API calls | None | Per-request to provider |
| Cost | Zero | Token-based billing |
| Latency | Microseconds | 1-5 seconds |
| Deterministic | Yes | No |
| Auditable | Line-of-code traceable | Prompt-dependent |
| Offline capable | Yes (with cached data) | No |
| Test coverage | 34,000+ unit tests | Integration tests |
| Required for dashboard | Yes | No |

The insight is this: **most care home intelligence does not need generative AI**. It needs structured data, domain-specific rules, and clear presentation. Cara achieves this with pure functions, and reserves LLM capabilities for the genuinely unstructured tasks where they add real value.
